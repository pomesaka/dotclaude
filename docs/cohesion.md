# 凝集度

凝集度とは、モジュール内の要素がどれだけ単一の目的に集中しているかの指標。凝集度が高いモジュールは変更理由が1つしかなく、変更の影響範囲が小さい。

凝集度と結合度は表裏一体。凝集度を上げると、モジュール間の依存が自然に減る。

---

## 凝集度の7段階（強い順）

Larry Constantine・Edward Yourdon が定義した分類。上ほど良い。

| レベル | 名称 | 結合軸 |
|--------|------|--------|
| 7 | **機能的**（Functional） | 単一目的。全要素が1つの明確なタスクに貢献 |
| 6 | **逐次的**（Sequential） | 前段の出力が次段の入力になるデータフロー |
| 5 | **通信的**（Communicational） | 同じデータを操作する処理をまとめる |
| 4 | **手続き的**（Procedural） | 実行順序は決まっているがデータは非共有 |
| 3 | **一時的**（Temporal） | 同タイミングで実行する（`initialize()` に詰め込む等） |
| 2 | **論理的**（Logical） | 「同カテゴリ」という理由でまとめ、フラグで分岐 |
| 1 | **偶発的**（Coincidental） | 無関係。`Utils` クラスや `helpers.ts` がこれ |

目指すのは機能的凝集。「この関数の目的を一文で説明するとき接続詞が必要になるか」がチェックの基準。

---

## 関数レベルのチェック

| チェック | レッドフラグ |
|---------|------------|
| **「and」テスト** | 目的を一文で説明するとき「〜して、〜する」と接続詞が必要 |
| **抽象レベルの混在（SLAP）** | 高レベルのオーケストレーションと低レベルの詳細が同居している |
| **フラグ引数** | `boolean` パラメータで振る舞いを切り替えている → 2関数に分割 |
| **名前と副作用の乖離** | `getX()` が DB に書く、`calculateX()` が外部 API を呼ぶ |
| **引数の数** | 4つ以上 → オプション型にまとめる。3つでも全てが目的に必要か確認 |

---

## クラス/ファイルレベルのチェック

| チェック | レッドフラグ |
|---------|------------|
| **export の棚卸し** | export されたシンボルが全て同じ概念に関連しているか。`parseDate`・`formatCurrency`・`validateEmail` が同居していたら偶発的凝集 |
| **ファイル名 vs 内容** | 内容を正確に表すファイル名が付けられるか。`utils`・`helpers`・`misc` しか思いつかない場合、凝集度が低い |
| **変更理由の数** | 「このファイルを変更する独立したビジネス上の理由はいくつあるか？」2つ以上なら分割の候補 |
| **内部グルーピング** | ファイル内の関数を互いに参照しないグループに分割できるなら、ファイルを分割すべき |

**LCOM4**（Lack of Cohesion of Methods）：クラスのメソッド・フィールドをグラフとして表現し、連結成分数を数える。

```
LCOM4 = 1  → 凝集している（良い）
LCOM4 ≥ 2  → 独立した責務グループが存在 → 分割候補
```

## パッケージ/コンポーネントレベルの原則

Robert C. Martin（Clean Architecture）が定義した3原則。

### REP（Reuse/Release Equivalency Principle）
再利用の単位 = リリースの単位。「このパッケージを外部ライブラリとして出荷するとしたら、全要素が同梱されて意味をなすか？」

### CCP（Common Closure Principle）
同じ理由で同じタイミングに変わるものは同じパッケージへ。SRP のパッケージ版。
「機能 X を変更するとき、触るパッケージが5つある」→ CCP 違反。

### CRP（Common Reuse Principle）
一緒に使われないものは同じパッケージに置かない。ISP のパッケージ版。
「このパッケージの一部しか使わないのに、全体への依存が生じる」→ CRP 違反。

**テンション**：REP・CCP はパッケージを大きくする力、CRP は小さくする力。プロジェクト初期は CCP 優先（変化に追従）、安定フェーズで REP/CRP を強化（再利用性を高める）。

## アーキテクチャ/システムレベルの凝集度

差分だけでなく、**このコードはどこにあるべきか**を問う視点。

| チェック | 問い |
|---------|------|
| **パッケージ配置の妥当性** | 追加された関数・型・定数は意味的に正しいパッケージにあるか |
| **レイヤー整合性** | ドメイン層がインフラ層（DB・HTTP の詳細）に依存していないか |
| **ドメイン境界の侵食** | あるドメインのエンティティが別ドメインのビジネスロジックを含んでいないか |
| **概念の散乱** | 同じ概念が複数のパッケージで異なる形で再実装されていないか |
| **抽象の漏洩** | DB スキーマ名・API エンドポイント等の外部詳細がドメイン層まで滲み出ていないか |

**Bounded Context（DDD）**：コンテキスト内は高凝集（共通の言語・ルール・モデル）、コンテキスト間は疎結合（明示的なインターフェースのみ）。別コンテキストの用語・モデルが混入していないか確認する。

**Conway's Law**：凝集度の高いサービス境界はチーム境界と一致する。チームをまたぐコンポーネントは凝集が下がりやすい。

## 結合度の指標と依存原則

凝集度を上げると結合度は自然に下がる。以下は結合度の定量的な見方と原則。

```
Ca（求心性）= このコンポーネントに依存してくる数
Ce（遠心性）= このコンポーネントが依存している数
不安定度 I  = Ce / (Ca + Ce)  ← 0=安定、1=不安定
```

| 原則 | 意味 | レビュー観点 |
|------|------|-------------|
| **ADP**（非循環依存） | 依存グラフに循環なし | この変更で循環が生まれないか |
| **SDP**（安定依存） | 不安定なものは安定したものに依存する | 頻繁に変わるものへの依存がないか |
| **SAP**（安定抽象） | 安定したものは抽象的であるべき | 安定コンポーネントがインターフェース主体か |

## コードスメル

### Feature Envy（特性の横取り）
自モジュールより他モジュールのデータを多用する関数。そのコードは本来あるべき場所に移動すべき。

### Divergent Change（発散的変化）
1つのファイルが複数の独立した理由で繰り返し変更される。SRP の直接違反。変更理由ごとに Extract Class で分割する。

### Shotgun Surgery（散弾銃手術）
1つの概念的な変更のために多数のファイルを触る必要がある。Divergent Change の逆。CCP 違反。関連処理を1箇所に集約する。

### Data Clumps（データの群れ）
同じ複数フィールドの組み合わせが各所に繰り返し現れる。Value Object として凝集させる。

### Primitive Obsession（基本型への執着）
ドメイン概念をプリミティブで扱い、バリデーション・制約が散在する。Value Object に凝集させる。

### God Class
LCOM 高・インスタンス変数 20 超・行数 500 超が目安。Extract Class / Extract Subclass で分割する。

## Connascence（コネサンス）

Meilir Page-Jones が提唱した統一フレームワーク。凝集度と結合度を同一の概念で表現する。

> 「モジュール境界をまたぐコネサンスを最小化し、境界内のコネサンスを最大化せよ」

静的コネサンスの強度（弱い順）：名前 → 型 → 意味 → 位置 → アルゴリズム。
強いコネサンスがモジュール境界をまたぐとき、それは設計の問題を示すシグナル。
