# ドキュメンテーション・プラクティス

このドキュメントはドキュメンテーションの手法と、その目的を説明する。

## 目的

**新規参加者がコードを読む前に、ドメインの全体像を掴める状態を作る。**

コードは「何をしているか」を教えてくれるが「なぜそうなっているか」「どういう概念構造の上に成り立っているか」は教えてくれない。ドキュメントはこのギャップを埋める。

## 推奨する読み順

```
1. 00-glossary.md          まずドメイン用語を頭に入れる
2. 01-overview.md          システム全体像とデータフローを理解する
3. CLAUDE.md               開発規約とキーバインドを確認する
4. architecture.md         パッケージ構成と初期化フローを把握する
5. concurrency.md          並行処理のルールを理解する
6. adr/                    設計判断の背景を知る（必要に応じて）
```

## 採用プラクティス

### 1. ドメイン用語集 (Ubiquitous Language)

**ファイル**: `00-glossary.md`

DDD (Domain-Driven Design) の基本プラクティス。コードと会話で使う言葉を統一し、認識のずれを防ぐ。

**ルール**:
- 型名・変数名・コメントはすべて用語集の言葉で書く
- 用語の意味が変わったら、コードとドキュメントを同時に更新する
- 「この言葉はコードのどこに対応するか」を明記する

**参考**: Eric Evans "Domain-Driven Design" Chapter 2 "Communication and the Use of Language"

### 2. C4 モデル (Simon Brown)

**ファイル**: `01-overview.md`

4段階のズームレベルでアーキテクチャを記述する手法。このプロジェクトでは上位3レベルを使う:

| レベル | 対象 | ドキュメント |
|--------|------|-------------|
| **Context** | システム全体と外部アクター | `01-overview.md` |
| **Container** | プロセス・データストア | `01-overview.md` |
| **Component** | パッケージ間依存と責務 | `architecture.md` |
| Code | 型・メソッド | コード自体 + godoc |

**ルール**:
- Context/Container は「コードを読む前に読む」ドキュメント
- Component は「コードを読みながら参照する」ドキュメント
- Code レベルはコード自体に任せる（ドキュメントに型定義を複製しない）

**参考**: https://c4model.com/

### 3. データフロー図

**ファイル**: `01-overview.md` 内

複数のデータソースが1つのエンティティに合流する「投影 (projection)」アーキテクチャを持つシステムでは、データフロー図で明示する。

**ルール**:
- データの方向（どこから来てどこへ行くか）を必ず示す
- 各データソースがエンティティのどのフィールドを所有するかを明記する

### 4. ADR (Architecture Decision Records)

**ディレクトリ**: `docs/adr/`

設計判断の「なぜ」を記録する。コードは「何」を伝えるが「なぜそうしなかったか」は伝えない。

**採用フォーマット**: [MADR](https://adr.github.io/madr/) (Markdown Any Decision Records) の簡略版

```markdown
# ADR-NNN: タイトル

## ステータス
Accepted / Superseded by ADR-XXX / Deprecated

## コンテキスト
何が問題だったか、どういう制約があったか。

## 決定
何を選んだか。

## 結果
良い点、悪い点、トレードオフ。
```

**ADR を書くタイミング**:
- 複数の選択肢を検討して1つを選んだとき
- 既存の設計を意図的に変更したとき
- 「なぜこうなっているか」を将来の自分/他人に説明する必要があるとき

**ADR を書かなくていいとき**:
- 唯一の自明な選択肢だったとき
- フレームワークやライブラリが強制する構造に従っただけのとき

**参考**: Michael Nygard "Documenting Architecture Decisions"

### 5. 設計原則ドキュメント

**ファイル**: `design-principles.md`

コーディング規約ではなく「設計時の思考プロセス」を記述する。何を重視し、何を意識して設計判断を行うかの指針。

## ドキュメント更新のルール

### いつ更新するか

- **用語の追加・変更**: 新しいドメイン概念を導入したら `00-glossary.md` を更新
- **パッケージ構成の変更**: `architecture.md` を更新
- **並行処理ルールの変更**: `concurrency.md` を更新
- **設計判断**: `adr/` に新しい ADR を追加
- **CLAUDE.md**: ビルド手順やキーバインドなど開発規約の変更時

### いつ更新しないか

- コードの内部実装の詳細（関数の引数や戻り値の説明など）→ コードコメントに任せる
- 一時的な実装状態 → ADR で方向性だけ記録し、詳細はコードに任せる

### Living Documentation の原則

- ドキュメントは「正しいか古いか」の二択。「だいたい合っている」は許容しない
- コード変更とドキュメント更新は同じコミットで行う
- ドキュメントに書いた概念がコードに存在しなければ、ドキュメントを消す
- コードに存在する重要な概念がドキュメントに無ければ、ドキュメントを追加する
