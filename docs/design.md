# 設計品質

linterでは検出できない、人間が判断すべき設計上の問題。

---

## 1. 設計原則の基礎（SOLID）

SOLID は「変更しやすいコード」を目指す5原則。個別のルールとして暗記するより、**変更コストを下げる**という目的から原則を理解する方が実践的。

### Single Responsibility Principle（SRP）

モジュールが変更される理由は1つだけであるべき。「変更理由」はビジネス上の関心事（アクター）の単位で考える。

| 違反パターン | 症状 |
|------------|------|
| God Object / God Class | 1クラスが認証・永続化・ビジネスロジックを全部持つ |
| UserProfileManager | 解析・バリデーション・保存を1クラスが担う |
| Divergent Change | 同じファイルが無関係な理由で繰り返し変更される |

「この関数/クラスを一文で説明するとき `〜して、〜する` と接続詞が必要になるか」がチェックの基準。

### Open/Closed Principle（OCP）

拡張に対して開いており、修正に対して閉じている。新機能を追加するとき、既存コードを変更せずに済む構造。

| 違反パターン | 症状 |
|------------|------|
| Switch/if-else の列挙 | 型判定のswitch文があり、新種別追加のたびに全箇所を修正する |
| フラグ引数の増殖 | booleanパラメータで振る舞いを切り替える `process(data, true, false)` |

ポリモーフィズム・ストラテジーパターン・プラグイン構造が典型的な対策。ただし**変化点が実証される前に OCP を適用する（YAGNI 違反）のも問題**。

### Liskov Substitution Principle（LSP）

サブタイプはスーパータイプの契約（事前条件・事後条件・不変条件）を守る。

| 違反パターン | 症状 |
|------------|------|
| `throw NotImplementedException` | 継承してメソッドを無効化する |
| 事前条件の強化 | 基底クラスが null を受け付けるのに派生クラスで拒否する |
| 振る舞いの変質 | 正方形は長方形のサブタイプに見えるが、`setWidth` が高さも変えると契約違反 |

違反が疑われたら**継承を組み合わせ（コンポジション）に置き換える**。

### Interface Segregation Principle（ISP）

クライアントが使わないメソッドへの依存を強制しない。大きなインターフェースを目的ごとに分割する。

| 違反パターン | 症状 |
|------------|------|
| 肥大インターフェース | `IRepository` が CRUD に加えて検索・一括処理・監査ログまで持つ |
| 空実装 | インターフェースを実装してもほとんどのメソッドが空か例外を投げる |

### Dependency Inversion Principle（DIP）

上位モジュールは下位モジュールの具象に依存しない。どちらも抽象（インターフェース）に依存する。

| 違反パターン | 症状 |
|------------|------|
| 具象依存 | `UserService` が `MySQLUserRepository` を直接 `new` する |
| 変更カスケード | 下位の実装変更が上位まで波及する |

DIP の恩恵はテスタビリティにも直結する（→「7. テスタビリティ」節参照）。

## 2. シンプルさの原則（YAGNI・KISS・DRY）

### YAGNI（You Aren't Gonna Need It）

「いつか必要になるかも」で作る機能・抽象・汎用化は作らない。実際に変化が来てから設計を変える。

実装が1つしかないのに抽象化する、設定項目を作るが誰も使わない、これらは YAGNI 違反の典型。

### KISS（Keep It Simple, Stupid）

動く最もシンプルな解を選ぶ。複雑さを正当化するには、その複雑さが解決する問題が実在する必要がある。

パターン・カーゴカルト（Visitor パターン、Service Locator など）を問題もないのに導入するのは KISS 違反。

### DRY（Don't Repeat Yourself）と過剰適用

DRY は「知識の重複を避ける」原則であり、「コードの重複をすべてなくす」ではない。

**Sandi Metz の警告**: 「*間違った抽象はコードの重複よりずっと高くつく*」  
典型的な劣化パターン：
1. 重複を見つけて抽象を作る
2. 新しい要件が来て「ほぼ同じ」なので既存抽象に boolean パラメータを追加
3. さらに条件分岐が増え、誰も理解できない抽象になる

**Rule of Three**: 3回目の重複が来るまで抽象化を待つ。1〜2回の重複は「まだ方向性が見えていない」状態。

**WET（Write Everything Twice）の許容**: 異なるドメインの概念が偶然同じ形をしているだけなら、それぞれ独立して変化する。共通化すると変更理由が混在する。

## 3. 抽象化の品質

### 深い抽象 vs 浅い抽象（Ousterhout）

*A Philosophy of Software Design* のコアテーゼ：**良いモジュールはシンプルなインターフェースの裏に大きな機能を隠す**。

| 種別 | 定義 | 例 |
|------|------|----|
| 深い抽象 | インターフェースのコストに対して隠蔽する複雑さが大きい | Unix ファイル I/O（open/read/write だけでファイルシステム全体を隠す） |
| 浅い抽象 | インターフェースが複雑な割に機能が少ない | 1メソッドをラップするだけのクラス |

「クラスは小さくあるべき」という命題は浅い抽象を量産する。小ささより**情報隠蔽の量**が指標。

### 実装漏洩（Leaky Abstraction）

Joel Spolsky の法則：「すべての非自明な抽象は、ある程度漏洩する」。

| 漏洩パターン | 具体例 |
|------------|--------|
| 内部構造を知らないと使えない | ページネーション API でオフセット計算を呼び出し側に求める |
| パフォーマンス特性の漏洩 | SQL の等価クエリが実行計画によって 1000 倍速度差が出る |
| 例外の漏洩 | DB 固有の例外（SQLException）がドメイン層まで到達する |

対策は「漏洩ゼロを目指す」ではなく、「漏洩の種類と程度を把握して文書化する」。

### 正直な抽象

インターフェースが約束する以上の副作用が実装に隠れていないか。

- `getUser()` が DB に書き込む
- `calculateTotal()` が外部 API を呼ぶ
- `isValid()` がオブジェクトの状態を変更する

名前から予期できない副作用は、インターフェースに対する嘘であり、呼び出し元での推論を壊す。

## 4. API デザイン

### 誤用しにくい API（Making APIs Hard to Misuse）

Joshua Bloch の原則：「*よいことを簡単に、悪いことを難しく（できれば不可能に）する*」。

| アンチパターン | 問題 | 改善方向 |
|-------------|------|---------|
| Boolean 引数 | `process(data, true, false)` — 何が true か読めない | 名前付きオプション型、別関数に分割 |
| 順序が混同しやすい引数 | `copy(src, dst)` vs `copy(dst, src)` | 型を区別、オプション型にまとめる |
| `null`/`undefined` デフォルト | 暗黙の無効値が伝播する | 明示的な型で無効状態を表現 |
| 動作を変える暗黙のデフォルト値 | デフォルトが安全でない設定になっている | 安全なデフォルトを選ぶ（Secure by Default） |

### Principle of Least Surprise

「メソッドはその名前から期待される最も驚きの少ない動作をすべき」（Bloch）。

- `add()` がコレクションを変更せず新しいものを返す場合は `with()` や `plus()` などの名前に
- `save()` が例外を黙って飲み込まず、失敗を呼び出し元に伝える
- 副作用の有無が名前から自明でない場合はドキュメントで明示する

### 明示的 vs 暗黙的

| 暗黙的（避ける） | 明示的（推奨） |
|----------------|-------------|
| グローバル状態への依存 | 依存を引数/コンストラクタで渡す |
| 呼び出し順序への依存（`init()` が必要） | 構築時に完全な状態を作る |
| コンテキストオブジェクトの多用 | 必要な値だけを引数に取る |

### 入力バリデーション

システム境界（ユーザー入力・外部 API・RPC の受信側）でバリデーションを行い、内部モジュールには正規化済みのデータが来ることを保証する。内部で何度も同じバリデーションを繰り返すのは凝集度の低下であり、「誰がいつバリデーションするか」が不明確になる。

## 5. エラーハンドリングの設計

### 失敗モードは型で表現する

失敗しうる操作は戻り値の型でそれを可視化する。例外だけに頼ると、呼び出し元が失敗の可能性を静的に把握できない。

- Result 型 / Either 型（Rust: `Result<T, E>`、Haskell: `Either`、Go: `(T, error)` 多値返却）
- 判別可能ユニオン / Sum 型でエラー種別を網羅的に表現
- `throw` は「例外的な状況」に限定し、通常の制御フローに使わない

### 各層でのエラー変換

| 層 | 責務 |
|----|------|
| インフラ層 | 技術固有の例外（I/O エラー、タイムアウト、DB エラー）をスロー |
| ドメイン層 | インフラ例外を受けず、ドメイン例外に変換して上位へ伝える |
| アプリケーション層 | ドメイン例外を受けてユースケースの文脈でのエラーに集約 |
| プレゼンテーション層 | アプリエラーをユーザーに提示できる形式に変換 |

「低レベル I/O エラーがユーザーにそのまま届く」はエラー変換の欠如。逆に「すべての例外を最上位でまとめてキャッチして捨てる」は原因追跡を不可能にする。

### エラーの伝播と変換の判断基準

- **変換すべきとき**: 抽象の境界を越える（インフラ→ドメイン）
- **再スロー（ラップして伝播）すべきとき**: この層では対処できず、上位が意思決定すべきとき
- **ここで処理すべきとき**: この層がリトライ・フォールバック・補償処理の責務を持つとき
- **黙って飲み込んではいけないとき**: ほぼ常時。空の catch は原則コメントで意図を説明する

### アクション可能なエラーメッセージ

「エラーが発生しました」は不十分。「何が失敗し、なぜ失敗し、何をすれば解決できるか」が分かるメッセージにする。

## 6. 不変性とステート管理

### Immutability の設計上の利点

| 利点 | 理由 |
|------|------|
| 推論しやすい | 値が変わらないので、どこから参照しても同じ結果 |
| スレッドセーフ | 共有可変状態がないので排他制御が不要 |
| 副作用の局所化 | 変更は新しい値の生成として表現され、変更箇所が明確 |
| テストが容易 | セットアップで作った状態が操作後に壊れない |
| 変更履歴の追跡 | 旧状態と新状態を比較できる（Redux のタイムトラベルデバッグなど） |

### 副作用の局所化

副作用（I/O、グローバル状態変更、外部通信）は「純粋なコア」と「命令的なシェル」に分離する考え方（Functional Core, Imperative Shell）。

```
[外部世界] → [命令的シェル] → [純粋なコア（副作用なし）] → [命令的シェル] → [外部世界]
```

純粋関数（同じ入力に必ず同じ出力、副作用なし）はテストにもフリーの依存注入を必要とせず、そのまま単体テストできる。

### 可変ステートのレッドフラグ

| パターン | 問題 |
|---------|------|
| 遠距離作用（Action at a Distance） | A を変更したら B の動作が変わる（共有可変状態経由） |
| 順序依存 | 操作を特定の順で呼ばないと正しく動かない |
| フラグの組み合わせ爆発 | 複数の boolean フラグが複雑な状態マシンを形成している |

状態が複雑になるときは、**有効な状態のみを表現できる型（状態機械として型で表す）**を検討する。

## 7. テスタビリティと設計

### テストしにくいコードは設計の問題

「テストが書けない」は多くの場合、設計の問題のシグナル。

| テストしにくい原因 | 設計上の問題 | 改善方向 |
|-----------------|------------|---------|
| グローバル状態に依存 | 暗黙の依存、副作用 | DIP で依存を注入する |
| 具象クラスを直接 `new` | 高結合 | インターフェース + DI |
| 時刻・乱数・ファイルシステムへの直接参照 | テスト不能な副作用 | 抽象化して注入 |
| 巨大なセットアップが必要 | 責務が多すぎる（低凝集） | SRP に沿って分割 |
| テスト後に状態をリセットしなければならない | 副作用が広範囲 | 副作用を局所化 |

### Seam（縫い目）の考え方（Michael Feathers）

テストを可能にするには「縫い目（Seam）」—変更を加えずに振る舞いを差し替えられる点—が必要。代表的な縫い目：

- **ポリモーフィック Seam**: インターフェースを介して実装を差し替える（最も汎用）
- **パラメータ Seam**: 依存を引数として受け取り、テスト時にテストダブルを渡す
- **関数 Seam**: 高階関数で処理を注入する

Seam がないコードは変更しないとテストできない（テストのためだけの変更はリスク）。設計段階から Seam を考慮するのが本来の対策。

### 依存性注入（DI）の設計指針

DI は「テストのための技法」ではなく「依存を明示する設計原則」。

- コンストラクタ注入を基本とする（必須依存を明示）
- メソッド注入は「この呼び出しだけ異なる実装を使いたい」場面に限定
- DI コンテナ（フレームワーク）への依存は構成ルート（Composition Root）に集中させる

純粋関数は依存注入なしでテスト可能。副作用が多い層（インフラ）と少ない層（ドメインロジック）を分けると、テスト対象の大半が DI 不要になる。

### テストしやすいコードの特性

- **小さい関数**: 入力と出力が明確で、内部状態が少ない
- **明示的な依存**: 何に依存しているかが関数シグネチャから分かる
- **副作用の分離**: 純粋な計算と I/O が混在していない
- **決定論的**: 同じ入力に対して常に同じ結果を返す
- **単一責務**: テストが1つの概念を検証できる粒度になっている
