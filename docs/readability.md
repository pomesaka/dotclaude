# 可読性

可読性の目標は、コードの意図が読み手に素早く正確に伝わること。

## なぜ可読性が重要か（認知負荷の観点）

人間のワーキングメモリは同時に保持できるチャンクが概ね 4〜5 個に限られる。以下はそれを圧迫する典型的な原因：

- **不明確な名前**: 変数の意味を都度思い出す必要がある
- **深いネスト**: 現在どの条件分岐の中にいるかを追い続ける
- **抽象レベルの混在**: 「この関数は何をしているか」を再解釈し続ける
- **意外な副作用**: 「この呼び出しで何が変わるか」を常に疑う
- **慣習の不一致**: コードベース内で `fetch`/`get`/`load`/`retrieve` が混在

良い可読性とは、**読み手のワーキングメモリを解放する**設計のこと。

## 命名

**具体性**: 名前はその要素が何であるかを正確に伝える。`data`、`result`、`tmp`、`info` などの汎用的な名前はモジュールスコープでは避ける。

**スコープと命名粒度**：

| スコープ | 命名の方針 |
|---------|-----------|
| 5行以内のループ変数 | `i`, `x` で十分 |
| 関数スコープの変数 | 概念が伝わる名前 |
| モジュールスコープ | 正確・具体的な名前 |
| 公開 API | 意図と制約まで名前に含める |

**一貫性**: コードベース内で同じ概念には同じ動詞を使う。`fetchUser`/`getUser`/`loadUser`/`retrieveUser` が混在すると、読み手は「違いは何か」を考えてしまう。

**関数名は動詞句**: `filterProductsByType`、`formatCurrency`。名前に `Manager`・`Processor`・`Handler` が含まれる場合、責務が多すぎる可能性がある。

**boolean には疑問の接頭辞**: `is`、`has`、`can`、`should` を付ける。肯定形を優先: `isEnabled` > `isNotDisabled`。変数名だけで `true`/`false` の意味が明確になること。

**単位・状態を名前に含める**: `timeoutMs`、`durationSec`、`maxRetries`、`rawInput`、`unsafeHtml`、`sanitizedHtml`。

**マジックナンバー・マジックストリングは名前付き定数に抽出する**: `if (status === 3)` → `const STATUS_COMPLETE = 3`。

## 制御フロー

**ガード節を先頭に置く**: 前提条件チェック・異常系を先に処理して早期リターンする。正常系が最も外側のインデントレベルに来るようにする。

```typescript
// 避ける: ネストが深く正常系が埋もれる
function process(user) {
  if (user) {
    if (user.isActive) {
      if (user.hasPermission) {
        // 正常系がここに
      }
    }
  }
}

// 推奨: ガード節で早期リターン
function process(user) {
  if (!user) return;
  if (!user.isActive) return;
  if (!user.hasPermission) return;
  // 正常系がここに（フラットに読める）
}
```

**ネストは3段階まで**: それ以上は関数の抽出・早期リターン・条件の反転で平坦化できないか検討する。

**条件の書き方**:
- 肯定条件を先に: `if (isValid)` > `if (!isValid)`（両分岐がある場合）
- 短い分岐を先に: 2行のケースを先に処理すれば全体が見渡せる
- 変数を左辺に: `if (age >= 18)` > `if (18 <= age)`
- 三項演算子は単純な単一式のみ。ネストした三項演算子は禁止

## 関数設計

**単一抽象レベルの原則（SLAP）**: 関数内のすべての文は同じ抽象レベルであるべき。

```typescript
// 避ける: 「ユーザーを認証する」と「ヘッダーのバイト4-8を検証する」が同居
function authenticate(request) {
  const header = request.headers['authorization'];
  const bytes = Buffer.from(header, 'base64');
  const tokenPart = bytes.slice(4, 8);  // 低レベル
  const user = database.findUser(tokenPart);  // 高レベル
  if (!user.isActive) throw new Error('...');  // 高レベル
}

// 推奨: 抽象レベルを統一
function authenticate(request) {
  const token = extractToken(request);  // 低レベルを隠蔽
  const user = findUserByToken(token);
  assertUserIsActive(user);
}
```

**Command-Query 分離（CQS）**: 関数は「何かを行う（副作用）」か「何かを返す（問い合わせ）」のどちらか一方にする。`checkPassword()` がセッションも初期化してはいけない。「質問したら答えが変わる」状況は混乱の元。

**引数設計**:
- 4つ以上はオプション型にまとめる
- boolean 引数は関数を2つに分割する合図
- 引数の数が多いほど、関数の凝集度が低い可能性がある

**戻り値型を明示する（公開関数）**: 型推論できても、公開 API の戻り値型はドキュメントとして機能し、意図しない型変更を防ぐ。

## ファイル内の構成

**局所性の原則（Locality of Behaviour）**: 関連するコードを近くに置く。コードの動作を理解するために複数のファイル・スクロールを必要としない構造が理想。

**新聞メタファー（トップダウン）**: ファイルの先頭に見出し（export）、下に詳細（内部ヘルパー）を配置する。呼び出し元を上、呼び出し先を下に置く。

**変数は使用箇所の近くで宣言する**:

```typescript
// 避ける: 関数の先頭でまとめて宣言
function process() {
  const config = loadConfig();
  const logger = createLogger();
  // ... 20行後 ...
  logger.log('start');
}

// 推奨: 使う直前に宣言
function process() {
  const config = loadConfig();
  // configを使う処理
  const logger = createLogger();
  logger.log('start');
}
```

**空行を段落区切りとして使う**: 関連する行をまとめ、異なるまとまりの間に空行を入れる。

**一貫性**: ベストプラクティスより劣っていても、コードベース内で一貫している方が可読性は高い。慣習を変えるときはファイル全体・コードベース全体で統一する。

## コメント

**WHY を書く、WHAT は書かない**: コメントはコードが「なぜ」そうなっているかを説明する。コードが「何を」しているかの説明が必要なら、まずコード自体をリファクタリングする。

Ousterhout の見方：「コメントは抽象設計の道具。コードからは読み取れない高レベルの意図や制約を伝える。」

**書くべきコメント**:

| 種類 | 例 |
|------|-----|
| 設計判断の理由 | なぜこの方式を選んだか、却下した代替案 |
| 意図的なトレードオフ | パフォーマンスのために可読性を犠牲にした箇所 |
| 非直感的な制約 | 「この順で呼ばないと〇〇が壊れる」「スレッドセーフでない」 |
| 仕様外の挙動への対処 | 外部 API のバグや非公式な動作に依存する理由 |
| 意図的な空処理 | 空の catch ブロックが意図的である理由 |

**書くべきでないコメント**:

| 種類 | 理由 |
|------|------|
| コードの繰り返し | コードを読めば分かる（`// i を 1 増やす`） |
| 自明な関数への説明 | 名前が十分に説明的なら不要 |
| コメントアウトされたコード | バージョン管理が履歴を持つ |
| 変更履歴 | コミットメッセージの役割 |

**コメントの陳腐化**: コードを変更したらコメントも更新する。嘘をついているコメントは、コメントがないより有害。
