# テクニカルライティング

技術ドキュメントの品質を評価する基準。`.md` ファイルの差分レビューと、変更に伴うドキュメント更新チェックに使用する。

## 文章品質のチェック

| チェック | レッドフラグ |
|---|---|
| **読者設定** | 誰向けかが不明。前提知識が示されていない |
| **能動態** | 受動態が多く主体が不明瞭（「〜される」「〜された」が多用されている） |
| **曖昧さ** | 「適切に」「場合によっては」「できるだけ」など判断を委ねる表現がある |
| **WHY の欠如** | ルール・決定事項の理由がない。「何を」「どう」だけで「なぜ」がない |
| **用語の揺れ** | 同じ概念を複数の名前で呼んでいる（例: "auth token" と "access token" が混在） |
| **冗長な表現** | 一文が長い（目安: 40字超）。接続詞でつながれた複合文が多い |
| **構造の欠如** | 手順なのにリスト・ステップ形式になっていない。見出しがなく塊になっている |

## ドキュメント種別ごとの必須要素

### 仕様書 (spec / design doc)

| 項目 | 確認内容 |
|---|---|
| スコープ | 何を扱い、何を扱わないかが明記されているか |
| 前提条件 | 依存関係・制約条件が明示されているか |
| エッジケース | 境界値・異常系の挙動が記載されているか |
| 設計の根拠 | なぜこの設計を選んだかが書かれているか（代替案の棄却理由含む） |

### 運用手順書 (runbook / ops)

| 項目 | 確認内容 |
|---|---|
| 再現可能性 | 手順通りに実行すれば誰でも同じ結果になるか |
| コマンドの完全性 | 実行するコマンドが具体的・完全に書かれているか（省略なし） |
| 失敗時の対処 | エラーが起きたときの対処方法が書かれているか |
| 事前・事後確認 | 実行前の確認事項・実行後の正常確認方法があるか |

### コーディングポリシー (CLAUDE.md / rules / CODING_CONVENTIONS)

| 項目 | 確認内容 |
|---|---|
| WHY | ルールの背景・理由が書かれているか |
| 例示 | 良い例・悪い例がセットで示されているか（複雑なルールほど必須） |
| 例外 | 例外ケースとその対処方法が定義されているか |
| スコープ | どのファイル・どのチームに適用されるかが明確か |

## コードスメル

**Orphaned Doc**: コードは変更されたが対応するドキュメントが更新されていない。実装と乖離したドキュメントは読者を誤った方向に誘導する。

**TBD Accumulation**: 「後で書く」「要確認」が残ったままマージされる。未完成のドキュメントは存在しないよりも悪いことがある。

**Assumption Dump**: 書いた人だけが知っている前提が暗黙のまま。新規メンバーが読んでも文脈を理解できない。

**Changelog Neglect**: 破壊的変更・仕様変更があったにもかかわらず、変更履歴や関連ドキュメントが更新されていない。

## ドキュメント網羅性の判定基準

プロジェクトに Documentation Map が定義されている場合、差分に含まれる変更の種類に対応するドキュメント更新が揃っているかを確認する。

コードの変更だけがあってドキュメントが一切更新されていない場合は、変更内容が「ドキュメント不要な変更」（リファクタリング、バグ修正など）かどうかを判断した上で指摘する。明らかに仕様変更・機能追加・手順変更である場合は必ず指摘する。
