# setup-docs: ドキュメント体制の整備

あなたの目的は、プロジェクトのドキュメント構造を分析し、**Documentation Map** を生成して適切なファイルに追記することです。今すぐ Step 0 から開始してください。

---

## Step 0: プロジェクト分析

### プロジェクト名の把握
`jj git remote list` でリポジトリ名を確認する。

### ディレクトリ構造の把握
トップレベルのディレクトリ一覧を確認する。ドキュメント関連のディレクトリ（`docs/`, `doc/`, `documentation/` など）を特定する。

### 既存ドキュメントの確認
以下を存在する場合に Read する:
- `./CLAUDE.md`
- `./docs/README.md`
- `./docs/` 配下のファイル一覧（find コマンド等）
- `./README.md`

**すでに `## Documentation Map` セクションが存在する場合**: その内容を把握し、Step 1 で更新・補完する方針で進める。

### ソースコード構造の把握
どんな種類の変更が起きうるかを推定するため、主なサブシステムを把握する:
- `backend/` `server/` `api/` → バックエンド・API変更の可能性
- `frontend/` `web/` `app/` → フロントエンド変更の可能性
- `infra/` `terraform/` `k8s/` → インフラ変更の可能性
- `*.go` `*.ts` `*.py` など主要言語の分布

---

## Step 1: ドキュメントマップの草案作成

Step 0 の分析結果をもとに、以下の2つのテーブルを草案として作成する。

### ドキュメント一覧テーブル

プロジェクトに存在するドキュメントと、今後追加すべき重要なドキュメントを列挙する。

```markdown
| ファイル/ディレクトリ | 種別 | 目的 |
|---|---|---|
| docs/spec/ | 仕様書 | 機能仕様・設計ドキュメント |
| docs/runbook/ | 運用手順書 | デプロイ・障害対応手順 |
| CLAUDE.md | コーディングポリシー | 開発規約・アーキテクチャルール |
```

種別は `仕様書` / `運用手順書` / `コーディングポリシー` / `API仕様` / `ADR` / `README` から選ぶ。

### 変更→ドキュメント更新マッピングテーブル

どんな変更があったときにどのドキュメントを更新すべきかを定義する。プロジェクトの技術スタックと構造に合わせて具体的に書く。

```markdown
| 変更の種類 | 更新が必要なドキュメント |
|---|---|
| APIエンドポイント追加・変更 | docs/spec/api.md |
| 新機能追加 | docs/spec/ 配下の該当仕様書 |
| インフラ・デプロイ変更 | docs/runbook/ 配下の該当手順書 |
| コーディングルール変更 | CLAUDE.md |
| 環境変数追加・変更 | docs/runbook/env.md または README.md |
```

---

## Step 2: ユーザー確認

草案をユーザーに見せる前に、以下を自分で判断できない場合は AskUserQuestion で確認する:

- **書き込み先**: `CLAUDE.md` と `docs/README.md` の両方が存在し、どちらが適切か判断できない場合
- **ドキュメントの目的**: 既存 `.md` ファイルが何のためのものか不明な場合
- **変更マッピングの不明点**: このプロジェクト固有の変更種別があるか不明な場合

判断できる場合は確認せずに進む。

---

## Step 3: 書き込み先の決定

以下の優先順位で書き込み先を決定する:

1. `./docs/README.md` が存在する → そのファイルに追記
2. `./CLAUDE.md` が存在する → そのファイルに追記
3. どちらもない → `./CLAUDE.md` を新規作成して書き込む

---

## Step 4: Documentation Map の書き込み

決定した書き込み先に以下のセクションを追記する（既存の `## Documentation Map` セクションがある場合は置き換える）。

Edit ツールを使って追記する。ファイル末尾に追加する場合は、最後の内容との間に改行を1行入れる。

```markdown
## Documentation Map

プロジェクトで維持すべきドキュメントと、変更種別ごとの更新要件。
`/setup-docs` で生成・更新できる。

### ドキュメント一覧

| ファイル/ディレクトリ | 種別 | 目的 |
|---|---|---|
（Step 1 の草案を埋める）

### 変更 → ドキュメント更新マッピング

| 変更の種類 | 更新が必要なドキュメント |
|---|---|
（Step 1 の草案を埋める）
```

---

## Step 5: 完了報告

以下を報告する:
- 書き込み先ファイル
- ドキュメント一覧（件数）
- 変更マッピング（件数）
- `docs-reviewer` を有効にするには `/kickoff-review-team` を実行するよう案内する
