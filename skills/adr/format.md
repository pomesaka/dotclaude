# ADR フォーマットテンプレート

## ファイル構造

```markdown
# ADR-NNNN: <決定のタイトル>

**Status:** Accepted
**Date:** YYYY-MM-DD

## Context

<なぜ今この決定が必要だったか。問題の背景と制約を書く。>

### 検討した選択肢

#### 1. <採用したアプローチ>（採用）

<説明>

#### 2. <却下した代替案>

<説明>

<必要に応じて特性比較テーブルや要件の箇条書きを入れる>

## Decision

**<採用したアプローチ> を採用する。**

<採用理由を 2〜4 箇条で書く。Context の特性と照らし合わせた理由が望ましい>

- **理由1**: 説明
- **理由2**: 説明
- **理由3**: 説明

## Consequences

### 受け入れた制約

- **制約1**: 何が犠牲になるか、どんな問題が起きうるか
- **制約2**: ...

### 再評価の条件

以下のいずれかに該当した場合、<別のアプローチ> への移行を検討する:

- <具体的・計測可能な条件1>
- <具体的・計測可能な条件2>
```

---

## Status の種類

| Status | 意味 |
|--------|------|
| `Proposed` | 提案中（レビュー待ち） |
| `Accepted` | 承認済み・採用中 |
| `Deprecated` | 廃止（新規では使わない） |
| `Superseded by ADR-NNNN` | 上位のADRに置き換えられた |

承認フローのないプロジェクトでは最初から `Accepted` で書いてよい。

---

## 書き方のベストプラクティス

### Context

- **過去形・現在形で書く**: 「〜だった」「〜である」— 決定当時の事実を記録する
- **問題を中立的に書く**: 「XXXが課題だった」ではなく「XXXという状況だった」
- **前提条件を明示する**: ペイロードサイズ、チーム規模、更新頻度など、決定に影響した数値や条件

### Decision

- **断言する**: 「〜を採用する」「〜を使う」— 曖昧にしない
- **短く**: 2〜3 文以内。長くなるなら理由を Context に移す
- **採用理由は Context の特性と対応させる**: Context に書いた特性が、なぜ Decision に繋がるかを示す

### Consequences

- **受け入れた制約は正直に**: ADR は正当化文書ではなく記録。デメリットを隠さない
- **再評価の条件は具体的に**: 「スケールしなくなったら」→「テーブル数が 100 を超えたら」
- **コードへの影響を書いてもよい**: 「`useRdbTable` の実装のみ変更すれば差し替えられる」のような実装ヒント

### ファイル命名

```
NNNN-<kebab-case-lowercase>.md
```

- 連番は必ず4桁ゼロ埋め
- タイトルは動詞始まりが読みやすい: `use-`, `adopt-`, `split-`, `remove-`
- 日本語ファイル名は避ける（git の文字化けリスク）

### 決定の粒度

ADR を書くべき判断の目安:
- **代替案が 2 つ以上あった**（= 迷った）
- **将来チームが同じ問いに直面しうる**
- **コードだけでは判断の理由が読み取れない**
- **トレードオフを明示的に受け入れた**

逆に書かなくていいもの:
- 慣例通りの決定（「React を使う」等）
- コードコメントで十分なレベルの局所的判断
- まだ仮説段階の設計（採用確定後に書く）
