## このスキルの目的

コードを読んで「なぜこうなっているか」を逆算する。

通常の設計作業の逆方向：

```
coding 方向: mental model ──→ design doc ──→ code
review 方向: code ──────────→ design doc ──→ mental model（推測）
```

C4 モデルの Component レベルの構造と、ADR 的な「なぜ」の推測を出力する。

## 解析ステップ

### 1. 対象の把握

上記「解析対象」の値に厳密に従い、指定範囲を超えてファイルを読まない。

| 入力パターン | 調査対象 |
|---|---|
| `pr <N>` | `gh pr view <N> --json files` でファイル一覧取得 → `gh api .../pulls/<N>/files` で各ファイルの diff を取得。**差分があったファイルのみ**を読む |
| `jj diffu -r 'main..@'` 等の jj/git diff コマンド | そのコマンドを実行して変更ファイル一覧を取得 → **差分があったファイルのみ**を読む |
| `<path>` （ファイルまたはディレクトリ） | 指定パス配下のファイルのみを読む。親ディレクトリや兄弟ディレクトリには立ち入らない |
| 指定なし（空） | エラー: 解析対象が不明確なため、ユーザーに対象を確認する |

### 2. コードの読み込み

優先順位：
1. **型定義・インターフェース** — 何を表しているかの骨格
2. **エントリーポイント**（公開 API / index.ts / main 等）— 全体像
3. **処理フローのキー関数** — データの流れ
4. **テスト** — 設計者の意図が最も正直に出る

### 3. Design Doc の組み立て（C4 Component レベル）

以下の問いに答えながら構造を整理する：

- 主要なコンポーネント（型・モジュール）は何か
- それぞれの責務は何か（RDD: 何を知っているか・何をするか・誰と協調するか）
- データはどこからどこへ流れるか
- 繰り返し現れる構造・規則性は何か

### 4. Mental Model の抽出

Design Doc を1段抽象化して「このシステムは〇〇というメタファーで動いている」という一文を導く。

良い Mental Model の例：
- 「2つのスナップショットを比べて、住所付き変更チケットのリストを作る」
- 「LSP をプロトコル専任レイヤーに閉じ込め、ドメイン知識はその外側に置く」

Mental Model は Design Doc の要約ではない。抽象度が1段上の概念フレーム。

### 5. ADR 候補の推測

コードから読み取れる「設計判断の痕跡」を ADR 形式で推測する：
- なぜこの構造になっているか（制約・トレードオフの仮説）
- 何が犠牲になっているか
- 捨てられた可能性がある別の設計は何か

## 出力フォーマット

解析結果はそのままユーザーに見せること（スキルの返り値としてだけ返すのではなく、テキストとして直接出力する）。

```markdown
## Mental Model
> 「一文でシステムの動作原理を表す」

## Design Doc

### 構造
（C4 Component レベル：主要コンポーネントと責務・データフロー）

### 主要パターン
（繰り返し現れる設計パターン・規則性）

### 前提・制約
（この設計が成立するための前提条件）

## ADR 候補（推測）

### なぜこの設計か
（設計判断の背景仮説）

### トレードオフ
（この設計で犠牲になっているもの）
```

## Gotchas

- **指定範囲を超えてファイルを読む**: `pr <N>` なら差分ファイルのみ、`<path>` なら指定パスのみ。「関連ありそうなファイル」を自主的に広げない。設計の背景理解が不十分なら、出力の ADR 候補に「不明」と書く方が正直
- **コードの「何」を読もうとする**: 目的は「なぜ」の推測。型定義と命名から意図を読む。実装詳細より設計の骨格を優先する
- **Mental Model を Design Doc の要約にする**: Mental Model は抽象度が1段上。「コンポーネントが3つある」は Design Doc、「〇〇というメタファーで動く」が Mental Model
- **ADR 候補を省略する**: 「なぜ」の推測こそがこのスキルの核心。コードから読み取れる設計の痕跡を必ず書く
