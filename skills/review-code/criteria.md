# コードレビュー基準

linterで検出できない「構造・設計・可読性」の問題を人間がレビューで捉えるためのガイド。

## 凝集度

凝集度とは、モジュール内の要素がどれだけ単一の目的に集中しているかの指標。凝集度が高いモジュールは変更理由が1つしかなく、変更の影響範囲が小さい。

### 関数レベルのチェック

| チェック | レッドフラグ |
|---------|------------|
| **「and」テスト** | 関数の目的を一文で説明するとき「〜して、〜する」と接続詞が必要になる場合、責務が複数ある |
| **抽象レベルの混在** | 高レベルのオーケストレーション（`initializeApp`）と低レベルの詳細（`buffer.slice(0, 4)`）が同居している |
| **フラグ引数** | `boolean`パラメータで振る舞いを切り替えている → 2つの関数に分割すべき |
| **名前と副作用の乖離** | `getX()`や`calculateX()`がDBに書き込むなど、名前から予期できない副作用がある |
| **引数の数** | 4つ以上 → オプションオブジェクトにまとめる。3つでも全てが関数の目的に必要か確認する |
| **行数** | 30行超は調査対象。一文で要約できるかが基準で、行数は目安 |

### モジュール・ファイルレベルのチェック

| チェック | レッドフラグ |
|---------|------------|
| **exportの棚卸し** | exportされたシンボルが全て同じ概念に関連しているか。`parseDate`・`formatCurrency`・`validateEmail`が同居していたら偶発的凝集 |
| **ファイル名 vs 内容** | 内容を正確に表すファイル名が付けられるか。`utils`・`helpers`・`misc`しか思いつかない場合、凝集度が低い |
| **変更理由の数** | 「このファイルを変更する独立したビジネス上の理由はいくつあるか？」 2つ以上なら分割の候補 |
| **importのfan-out** | 無関係なモジュールから大量にimportしている → 複数の関心事をオーケストレーションしすぎ |
| **内部グルーピング** | ファイル内の関数を互いに参照しないグループに分割できるなら、ファイルを分割すべき |

### コードスメル（凝集度の低さを示す兆候）

**Feature Envy**: 関数が自身のモジュールのデータよりも他モジュールのデータを多く参照している。その関数は参照先に移動すべき。

**Shotgun Surgery**: 1つの概念的な変更が5つ以上の無関係なファイルに影響する。散らばった責務を1箇所に統合すべき。

**Divergent Change**: 1つのファイルが無関係な理由で繰り返し変更される。責務の境界に沿ってファイルを分割すべき。

**Data Clumps**: 同じ変数の組み合わせが複数の関数シグネチャに繰り返し現れる（例: `host, port, protocol`）。型として抽出すべき。

### 凝集度と結合度の関係

モジュール内の凝集度が高ければ、モジュール間の結合度は自然に低くなる。以下は結合度の問題を示す実用的なシグナル:

| シグナル | 問題 |
|---------|------|
| AモジュールをテストするのにBを常にモックする必要がある | Aの依存が広すぎる（低凝集→高結合） |
| モジュールAとBが相互importしている | 責務が混在している（循環依存） |
| 共有型を変更すると10以上のファイルが壊れる | 型が複数のドメインの関心事を混ぜている |

## 可読性

可読性の目標は、コードの意図が読み手に素早く正確に伝わること。

### なぜ可読性が重要か：認知負荷の観点

コードの読み手は「ワーキングメモリ」でコードを理解する。ワーキングメモリは同時に保持できるチャンク数に上限があり、研究によって概ね 4〜5 個が限界とされる。可読性の問題は、この上限を不必要に圧迫する設計になっている。

- **不明確な名前**: 読み手が名前の意味を推測・記憶するためにスロットを消費する
- **深いネスト**: 各インデントレベルを「今どの条件が成立しているか」としてスタックに積む必要がある
- **抽象レベルの混在**: 高レベルの文脈と低レベルの詳細を同時に保持しなければならない
- **意外な副作用**: 関数呼び出し後に「状態が変わったかもしれない」という不確実性をずっと保持し続ける
- **不一致な慣習**: 「この書き方は何を意味するのか」という解釈コストが毎回発生する

コードレビューでは、これらのコストが積み重なっていないかを確認する。

### 命名

**具体性とスコープの関係**: 名前の長さはスコープの広さに比例させる。

| スコープ | 基準 | 例 |
|---------|------|---|
| ループ変数・3行以内のラムダ | 短くて良い | `i`、`e`、`x` |
| 関数内ローカル変数 | 型か役割が分かる程度 | `user`、`total` |
| モジュールスコープ・export | 完全に自己説明的 | `activeUserCount`、`invoiceTotalBeforeTax` |
| 公開API・型定義 | 使用文脈が明確になるほど具体的 | `PaginatedUserListResponse` |

`data`、`result`、`tmp`、`info` はモジュールスコープでは避ける。読み手がその変数の意味を把握するために、定義箇所まで遡る必要が生じる。

**一貫性**: 同じ概念には同じ単語を使い続ける。`fetch` / `get` / `retrieve` / `load` を混在させると、読み手は「これらは同じ操作か、異なる操作か」を毎回推測するコストを払う。チームで決めた語彙から外れるときは明示的な理由が必要。

**関数名は動詞句、型名は名詞**:
- 関数: `filterProductsByType`、`formatCurrency`、`validateEmailAddress`
- 型: `OrderStatus`、`ProductItem`、`PaginationCursor`
- 名前に `Manager`・`Processor`・`Handler` が含まれる場合、責務が多すぎる可能性がある

**Boolean命名の原則**:
- `is`、`has`、`can`、`should` の接頭辞を付ける（読むと質問文になる）
- 肯定形を優先: `isEnabled` > `isNotDisabled`（二重否定は認知負荷が高い）
- 変数名だけで `true`/`false` の意味が明確になること: `shouldRetry`（trueなら再試行する）

**単位・状態を名前に含める**:
- `timeoutMs`、`durationSec`（単位の取り違えはサイレントなバグになる）
- `maxRetries`、`retryCount`（上限と現在値の区別）
- `rawInput`、`sanitizedHtml`、`unsafeUrl`（処理済みか否かの区別）

**マジックナンバー・マジックストリングは名前付き定数に抽出する**: `if (status === 3)` → `const STATUS_COMPLETE = 3`。数値・文字列リテラルが何を意味するかはコードから読み取れず、読み手は「3が何なのか」を仕様書や他のファイルで調べる必要がある。

### 制御フロー

**ガード節を関数の先頭に置く（早期リターン）**: 前提条件・異常系を先に処理してリターンし、メインロジック（ハッピーパス）を最も浅いインデントレベルに置く。

ガード節がない場合、読み手はすべてのネストを追いながら「正常系はどのパスか」を常に意識し続けなければならない。ガード節があれば、最初の数行で異常系を処理済みにでき、残りを「正常な状態」として読める。

```typescript
// 避ける: 全パスを追わないと正常系がどこか分からない
function process(user: User | null, order: Order | null) {
  if (user) {
    if (order) {
      if (order.status === 'pending') {
        // ここがメインロジック（深いネストの中に埋まっている）
      }
    }
  }
}

// 推奨: 異常系を先に排除し、以降は正常な状態として読める
function process(user: User | null, order: Order | null) {
  if (!user) return;
  if (!order) return;
  if (order.status !== 'pending') return;
  // メインロジック（ネストなし）
}
```

**ネストは3段階まで**: それ以上は関数の抽出・早期リターン・条件の反転で平坦化する。各ネストレベルは「このブロック内ではXという条件が成立している」という文脈をワーキングメモリに追加する。4段階以上では保持しなければならない文脈数がワーキングメモリの上限に近づく。

**条件の書き方**:
- 肯定条件を先に: `if (isValid) { ... } else { ... }` — 否定形の分岐を先に読まされると、「何でないか」から「何であるか」に変換するコストが発生する
- 短い分岐を先に: 短い処理を先に書くと、長い処理を読む前に「その条件の処理は終わった」と確定できる
- 変数を左辺に: `if (age >= 18)` > `if (18 <= age)` — 英語・日本語の語順と合致しているため直感的に読める
- 三項演算子は単純な単一式のみ。ネストした三項演算子は禁止（条件・真値・偽値の3つを同時に追わなければならず、ネストすると複雑度が指数的に増える）

### 関数設計

**単一抽象レベルの原則 (SLAP)**: 関数内のすべての文は同じ抽象レベルであるべき。

抽象レベルが混在すると、読み手は「この低レベルな処理が高レベルの文脈にどう対応するのか」を自分で補完しながら読まなければならない。この補完作業がワーキングメモリを圧迫する。

```typescript
// 避ける: 高レベル（認証）と低レベル（バイト操作）が混在
async function authenticateRequest(req: Request) {
  const token = req.headers.get('Authorization')?.slice(7);
  const headerBytes = Buffer.from(token ?? '', 'base64');
  const [header, payload] = headerBytes.toString().split('.');
  // ... JWT の低レベルパース処理 ...
  await db.query('SELECT * FROM users WHERE id = $1', [userId]);
  await session.initialize(userId);
}

// 推奨: 各行が同じ抽象レベル（認証フロー）
async function authenticateRequest(req: Request) {
  const token = extractBearerToken(req);
  const claims = verifyJwt(token);
  const user = await fetchUser(claims.userId);
  await initializeSession(user);
}
```

低レベルの処理を名前付き関数に抽出すると、読み手は高レベルの文脈を保ちながら「詳細を見たければ関数を辿る」という選択ができる。

**Command-Query分離 (CQS)**: 関数は「何かを行う（Command: 副作用）」か「何かを返す（Query: 問い合わせ）」のどちらか一方にする。

CQSが破られると「質問したら答えが変わる」状態が生まれる。読み手は関数を呼んだ後「状態が変わったかもしれない」という不確実性をずっと保持しなければならず、関数呼び出しのたびに副作用を確認するコストが発生する。

```typescript
// 避ける: getUser() を呼ぶとセッションが初期化される（予期しない副作用）
function getUser(id: string): User {
  const user = db.find(id);
  session.currentUserId = id;  // ← 副作用が隠れている
  return user;
}

// 推奨: Query と Command を分ける
function getUser(id: string): User { return db.find(id); }
function setCurrentUser(id: string): void { session.currentUserId = id; }
```

**引数設計**:
- 引数が 4 つ以上: オプションオブジェクトにまとめる。引数の順序を覚える必要がなくなり、名前付きで渡せる
- boolean 引数: 呼び出し側で `true` / `false` の意味が分からない。関数を 2 つに分けるか、判別可能ユニオンを使う
- 引数が増えるほど関数が複数の責務を持っていることが多い。引数数は凝集度の間接指標

**戻り値型を明示する（公開関数）**: TypeScript は型推論できるが、公開 API の戻り値型はドキュメントとして機能し、意図しない型変更をコンパイル時に検出できる。

### ファイル内の構成

**新聞メタファー（トップダウン）**: ファイルの先頭に見出し（export・公開インターフェース）、下に詳細（内部ヘルパー）を配置する。呼び出し元を上、呼び出し先を下に置く。

ファイルを開いたとき、最初に「このファイルが外に提供するもの」が目に入る。詳細実装を読む必要があるときだけ下にスクロールすれば良い。

**局所性の原則（Locality of Behavior）**: 宣言と使用を近くに置く。変数を使用箇所の近くで宣言し、関数の先頭でまとめて宣言しない。

宣言と使用が離れていると、読み手は変数の型・初期値・変更有無を確認するたびに宣言箇所まで遡る必要がある。特にスクロールが必要な距離になると、コンテキストを頭に保ちながら行き来するコストが発生する。

```typescript
// 避ける: 関数の先頭で全変数を宣言
function buildReport(data: RawData[]) {
  const filtered: RawData[] = [];
  const mapped: ProcessedData[] = [];
  const total: number;
  const result: Report;
  // ... 20行後 ...
  filtered = data.filter(isValid);
  mapped = filtered.map(transform);
  total = mapped.reduce((s, d) => s + d.value, 0);
  result = { items: mapped, total };
  return result;
}

// 推奨: 使用直前に宣言
function buildReport(data: RawData[]) {
  const filtered = data.filter(isValid);
  const mapped = filtered.map(transform);
  const total = mapped.reduce((s, d) => s + d.value, 0);
  return { items: mapped, total };
}
```

**空行を段落区切りとして使う**: 関連する処理をまとめ、異なるまとまりの間に空行を入れる。空行は「ここで話題が変わる」という視覚的シグナル。過剰な空行（意味のない空行）はリズムを壊す。

**一貫性**: コードベース内の慣習が一貫していることは、それ自体が可読性に貢献する。

慣習が一貫していると、読み手は「この書き方がここでは何を意味するか」を学習コストなしに即座に理解できる。慣習が混在していると、毎回「これはどちらのスタイルか」「特別な意味があるのか」を推測するコストが発生する。ある慣習がベストプラクティスより劣っていても、コードベース全体で一貫している方が可読性は高い（一貫性は予測可能性を生む）。

### コメント

**WHY を書く、WHAT は書かない**: コメントはコードが「なぜ」そうなっているかを説明する。コードが「何を」しているかを説明するコメントは、コードを読めば分かることを繰り返しているに過ぎない。WHAT コメントが必要と感じたなら、まずコード自体（命名・構造）をリファクタリングする。

Ousterhout が「コメントは抽象設計の道具」と述べるように、インターフェースコメントは「この関数の約束（契約）」を記述し、コードが自明でない「なぜ」を補完する役割を持つ。

**書くべきコメント**:

| 種類 | 例 |
|------|---|
| 設計判断の理由 | `// ライブラリXではなくYを使うのは、マルチバイト文字の処理が正確なため` |
| 意図的なトレードオフ | `// O(n²)だが入力サイズが常に小さいため（最大50件）最適化より可読性を優先` |
| 非直感的な制約 | `// DBドライバの制約でここでトランザクションをコミットする必要がある` |
| 警告 | `// この処理は最大30分かかる。UIにプログレス表示が必要` |
| 意図的な空処理 | `// エラーは呼び出し元に伝播させるため、ここでは何もしない` |
| 仕様外の挙動への対処 | `// APIが空配列でなくnullを返す場合があるため（仕様書 p.42参照）` |

**書くべきでないコメント**:

| 種類 | 理由 |
|------|------|
| コードの繰り返し（`// i を 1 増やす`） | コードを読めば分かる |
| 自己説明的な関数の上のコメント | 名前が十分に説明している |
| コメントアウトされたコード | バージョン管理が履歴を保持する |
| 変更履歴（`// 2024-01-15 Yamada 修正`） | コミットメッセージの役割 |
| TODO の放置 | 期限と担当者なき TODO は死蔵される |

**コメントの陳腐化**: コードが変更されてもコメントが更新されないと、誤情報を伝えるコメントになる。コメントは少なくすることで陳腐化リスクを下げる。コメントが必要な箇所が多いほど、コード自体の設計を見直すシグナル。

## 設計品質

linterでは検出できない、人間が判断すべき設計上の問題。

### 抽象の質

| 問い | レッドフラグ |
|-----|------------|
| この抽象は実装を漏洩していないか？ | 利用者が内部構造を知らないと正しく使えない |
| 実装は1つしかないのに抽象化していないか？ | 間接層のコストだけ増え、柔軟性の利点がない（YAGNI違反） |
| 正直な抽象か？ | インターフェースが約束する以上の副作用が実装に隠れている |

### エラーハンドリング

- **失敗モードは型で表現されているか**: 失敗しうる関数は戻り値型で失敗を可視化する（判別可能ユニオン、Result型など）。throwだけに頼らない
- **エラーは適切な層で処理されているか**: 低レベルI/Oエラーがユーザーにそのまま伝わってはいけない。各層が自分の抽象レベルにエラーを変換する
- **エラーメッセージはアクション可能か**: 「エラーが発生しました」は不十分。何が失敗し、何をすべきかが分かるメッセージにする

### TypeScript固有のパターン

**判別可能ユニオンでbooleanフラグを置き換える**:

```typescript
// 避ける: どのフィールドがいつ有効か不明確
type Response = { success: boolean; data?: Data; error?: string }

// 推奨: 型システムが有効な状態のみを許可
type Response = { success: true; data: Data } | { success: false; error: string }
```

**`readonly`の徹底**: 変更されないプロパティ・配列には`readonly`を付ける。意図しないmutationをコンパイル時に検出できる。

**`satisfies`の活用**: 型の検証とリテラル型の保持を両立したい場合に使う。`as`キャストの代替として有用。

### APIの誤用しにくさ

- boolean引数やあいまいなデフォルト値がないか
- パラメータの順序が混同しやすくないか
- 入力バリデーションはシステム境界（ユーザー入力、外部API）で行われているか
- 既存のAPI設計パターンと一貫しているか

## レビューコメントの書き方

### 良いコメントの3要件

1. **アクション可能**: 何をすべきか明確
2. **理由付き**: なぜその変更が必要か
3. **重大度に応じた表現**: 些細な指摘と重要な指摘を区別する

### 重大度ラベル

- **Nit:** 些細。ブロッキングではない。「Nit: `d`を`durationMs`にリネームすると意図が明確になります」
- **提案:** 採用は任意。「提案: 判別可能ユニオンを使うとシンプルになります」
- **(ラベルなし):** 対応必須のブロッキングイシュー

### 良いコメント例

```
この関数はusersが空配列のとき undefined を返します（find() の仕様）。
この値は formatUser() に渡されますが、undefined を処理しないため
実行時にthrowします。早期リターンか、not-found用の判別可能ユニオンを
検討してください。
```

失敗モード → 影響 → 修正案の順で説明する。
