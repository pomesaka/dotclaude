## レビューの進め方

### 1. ドメインモデルの健全性

対象ディレクトリ内のコードを読み、以下を分析する:

- **ユビキタス言語の一貫性**: 型名・関数名・変数名がドメインの概念を正確に反映しているか。技術用語がドメイン概念を隠していないか
- **Value Object の活用**: プリミティブ型の乱用（Primitive Obsession）がないか。ドメインの概念を表す値は専用の型になっているか
- **Entity の同一性**: ID で識別されるべきものが正しく Entity として表現されているか
- **Anemic Domain Model の兆候**: 型が getter/setter のみでロジックが外部の Service に流出していないか。ビジネスルールはそれを最もよく知る型に配置されているか

### 2. 責務の分析（RDD 観点）

各主要な型について:
- **何を知っているか**（Information Holder としての責務）
- **何をするか**（Service Provider / Controller としての責務）
- **誰と協調するか**（依存先は適切か）
- **責務が集中しすぎていないか**（God Object の兆候）
- **責務が薄すぎないか**（過剰な分割・不要な抽象化）

### 3. 境界とアーキテクチャ

- **Bounded Context の境界**: パッケージ/モジュールの境界がドメインの境界と一致しているか
- **依存の方向**: ドメインロジックが外部（UI, DB, フレームワーク）に依存していないか。依存性逆転の原則に従っているか
- **Context 間の連携**: 異なるコンテキスト間で直接の型参照がないか。変換層（ACL）や Domain Event で疎結合になっているか
- **集約の設計**: 集約の境界は適切か。不要に大きい集約はないか。他の集約を ID ではなくオブジェクト参照で保持していないか

### 4. 不変条件と型安全性

- **不正な状態の表現可能性**: 型システムで防げるはずの不正状態がランタイムチェックに頼っていないか
- **状態遷移の明示性**: 状態遷移が型で表現されているか（例: enum / sealed interface / 別の型）
- **エラーハンドリング**: ドメインのエラーがドメイン層で適切に表現されているか（インフラのエラーと混同していないか）

### 5. Refactoring Toward Deeper Insight

最も重要な観点:
- **現在のモデルに違和感はないか**: コードの「臭い」ではなく、ドメインの概念として不自然な構造はないか
- **隠れたドメイン概念はないか**: コメントやフラグで表現されている概念が、実はファーストクラスの型として表現すべきものではないか
- **より深い洞察の兆候**: 「この処理は実は○○というドメイン概念を暗黙的に扱っている」というパターンがあれば、それを明示的なモデルに昇格させる提案をする

## 出力形式

以下の形式でレビュー結果を報告する:

1. **ドメインモデルの概要**: 現在のモデル構造の要約
2. **良い点**: ドメインの理解が正確に反映されている箇所
3. **改善提案**: 具体的なファイルパスと行番号を含む改善案（優先度順）
4. **より深い洞察の候補**: 現在のモデルに暗黙的に存在する、明示化すべきドメイン概念

各提案には「なぜそれがドメインの理解を深めるか」の理由を添える。パターンの適用自体を目的にしない。

## Gotchas

- **差分レビューと混同する**: このスキルはドメイン設計の分析が目的。変更差分のコードレビュー（バグ・スタイル・ロジックの確認）は `review-code` を使う。
- **パターン適用を目的にする**: DDD/RDD の概念を押し付けない。「このパターンを適用すべき」ではなく「このドメイン概念を正確に表現するには」を軸に考える。提案にはドメインの理解がどう深まるかを必ず添える。
- **$ARGUMENTSを空にする**: 引数なしだと全コードが対象になり分析が浅くなる。分析したいパス（例: `internal/session/`）や観点を渡すと精度が上がる。
