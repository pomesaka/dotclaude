The user wants to create or review a Product Requirements Document (PRD).

## モード判定

ユーザーの発言から以下を判定する:

- **作成モード**: 「書いて」「作って」「作成して」など → Step 1: 要件ヒアリング
- **レビューモード**: 「レビューして」「確認して」「フィードバック」など → Step 2: PRDレビュー
- 判断できない場合は `AskUserQuestion` で確認する

---

## Step 1: PRD 新規作成

### 1-1. ヒアリング

`AskUserQuestion` で以下を確認する（まとめて一度に聞く）:

- **対象**: 何を作る？（機能 / プロダクト / MVP / 改善）
- **問題**: 誰のどんな問題を解くか
- **ゴール**: 成功したら何が変わっているか（定量指標があれば）
- **スコープ外**: 明示的に含めないもの

情報が不足していても作成は進める。仮置き項目は `[TBD: 理由]` で明記し、後で確認を促す。

### 1-2. PRD 構成

以下の構成で Markdown PRD を生成する:

```markdown
# [プロダクト/機能名] PRD

**Status**: Draft  
**Last Updated**: [日付]  
**Version**: 1.0

---

## Background

[背景・コンテキスト。なぜ今これを作るのか]

## Problem Statement

[解決する問題。誰が・何に困っているか。根本原因まで掘り下げる]

## Goals

- Goal 1: [SMART形式で書く]

## Success Metrics

| Metric | Baseline | Target | Measurement |
|--------|----------|--------|-------------|
| ...    | ...      | ...    | ...         |

## Target Personas

### [ペルソナ名]
- **Role**: ...
- **Pain Points**: ...
- **Goals**: ...

## User Stories

- As a [persona], I want to [action] so that [outcome]

## Features & Requirements

### Must Have (M)
- ...

### Should Have (S)
- ...

### Could Have (C)
- ...

### Won't Have (W)
- ...

## Out of Scope

- [明示的に含めないもの]

## UX / Design

[ワイヤーフレームのリンク or テキストでの概要]

## Assumptions & Constraints

- **Assumptions**: ...
- **Technical Constraints**: ...
- **Dependencies**: ...

## Timeline

| Milestone | Date | Owner |
|-----------|------|-------|
| ...       | ...  | ...   |

## Open Questions

- [ ] ...
```

### 1-3. 出力

- 保存先パスをユーザーに確認してから `Write` で保存する（デフォルト: `docs/prd/[feature-name].md`）
- 保存後、仮置き項目 `[TBD]` をリストアップして確認を促す

---

## Step 2: PRD レビュー

### 2-1. PRD の読み込み

- 引数でファイルパスが渡された場合はそれを `Read` で読む
- なければ `AskUserQuestion` でパスを確認する

### 2-2. レビュー基準

以下の観点で評価し、各項目を ✅ / ⚠️ / ❌ でスコアする:

| 観点 | チェック内容 |
|------|------------|
| **問題定義** | 根本問題が明確か。「機能がない」ではなく「何が困るか」を書いているか |
| **目標の SMART 性** | 具体的・測定可能・期限付きか |
| **成功指標** | KPI が定量的か。ベースラインと目標値があるか |
| **ユーザー定義** | ペルソナが具体的か |
| **スコープ明確性** | Out of Scope が書かれているか |
| **要件の書き方** | ソリューション（実装方法）ではなく要件として書かれているか |
| **優先度** | MoSCoW 等で優先度付けされているか |
| **未解決事項** | Open Questions が追跡可能か |
| **更新管理** | バージョン・変更履歴があるか |

### 2-3. フィードバック形式

```markdown
## PRD レビュー結果

### 総評
[2-3文で全体評価]

### 観点別チェック
[チェックリスト]

### 重要な改善点
1. **[問題点]**: [具体的な改善案]

### 改善案（差分）
[書き直し例があれば提示]
```

---

## Gotchas

- **ソリューションを要件と混同**: レビュー時に「ドロップダウンを追加する」のような実装指定は要件ではない。「ユーザーが複数オプションから素早く選択できる」のように書くよう促す
- **問題文が弱い**: 「〇〇機能がない」は問題文ではない。「〇〇がないことで発生する損失・摩擦」を書くよう促す
- **成功指標が定性的**: 「UX が改善される」は指標ではない。計測可能な数値に変換するよう促す
- **PRD 作成が Discovery の代替になっている**: ユーザーリサーチや検証なしに PRD だけ書いても意味がない旨を適宜コメントする
